# AgentSystem 多智能体系统审计报告

**审计日期**: 2026-02-13
**审计对象**: AgentSystem 项目
**对标框架**: LangGraph, AutoGen, CrewAI, Claude Agent SDK, OpenAI Agents SDK
**总体评级**: 原型阶段 (Prototype) | 行业标杆达成率: **38%**

---

## 执行摘要

AgentSystem 项目处于**早期原型阶段 (MVP)**。框架骨架已搭建，具备基本的多 Agent 串行协作能力，但在工业级标准的各个维度上都存在显著差距。

**核心发现**：
- 编排器硬编码为 Planner→Worker→Reviewer 三阶段，不支持动态路由
- Agent 通信靠字符串拼接，无结构化消息协议
- 零测试覆盖，无 pytest/CI/CD
- 工具调用依赖关键词匹配，未使用 LLM Function Calling
- 无真正的记忆系统和向量存储

### 维度 1: Agent 编排能力

**当前得分: 2.0 / 5**
**行业标杆: 4.5 / 5**
**差距: -2.5**
**改进优先级: P0 🔴**

#### 当前状态

✅ **已有能力**:
- 固定三阶段编排: Planner → Worker(s) → Reviewer（`orchestrator.py` 71-152行）
- 支持并行 Worker 执行（`run_async` 使用 `asyncio.gather`，154-230行）
- YAML 驱动的工作流定义（`router.py` 218-236行）
- 模板参数渲染 `{{variable}}`（`router.py` 203-216行）

❌ **缺失能力对比**:

| 能力 | LangGraph | AutoGen | CrewAI | 本系统 |
|------|-----------|---------|--------|--------|
| 动态工作流（运行时修改图） | ✓ StateGraph | ✓ | ✓ | ✗ |
| 条件分支/路由节点 | ✓ add_conditional_edges | ✓ | ✓ | ✗ |
| 循环/迭代（直到满足条件） | ✓ | ✓ | ✓ | ✗ |
| Supervisor/Router 模式 | ✓ | ✓ GroupChat | ✓ Manager | ✗ |
| Human-in-the-Loop | ✓ interrupt | ✓ | ✓ | ✗ |
| 子图/嵌套工作流 | ✓ subgraph | ✓ | ✓ | ✗ |
| 状态检查点/恢复 | ✓ checkpointer | 部分 | ✗ | ✗ |

#### 问题所在：硬编码的三阶段流水线

**代码证据** (`orchestrator.py` 71-152行):
```python
# 编排是固定的: planner → workers(全部) → reviewer
plan_resp = self.planner.run(planner_msg)

for worker in self.workers:
    worker_resp = worker.run(worker_msg, router=self.router)

reviewer_resp = self.reviewer.run(reviewer_msg)
```

**核心问题**:
1. **无条件分支**: 不能根据 Planner 输出动态决定调用哪些 Worker
2. **无循环机制**: Worker 执行失败后无法让 Planner 重新规划
3. **无回退能力**: Reviewer 发现问题后无法返回 Worker 重做
4. **拓扑固定**: 无法根据任务类型选择不同的协作拓扑（如 Supervisor、Swarm 等）

**示例场景说明问题**:
```
场景：用户问"分析最新的 AI 市场趋势"

当前系统执行流程：
1. Planner 规划 → "需要搜索资料、数据分析、撰写报告"
2. 所有 Worker 全部执行（即使某些 Worker 不适合这个任务）
3. Reviewer 审查 → 如果发现问题，只能重新执行整个流程

行业标杆（LangGraph）会这样做：
1. Planner 规划 → 决定先调用 "Research Agent"
2. Research Agent 执行 → 返回数据
3. 动态路由 → 根据数据类型决定调用 "Data Analyst" 或 "Content Writer"
4. Reviewer 发现问题 → 可以只让失败的 Agent 重试，无需全部重做
```

#### 改进建议

**短期（2周）**:
1. 实现条件边：Reviewer 可回退给 Worker
2. Planner 输出包含"需要调用哪些 Worker"的决策

**中期（1月）**:
1. 引入状态图抽象，参考 LangGraph StateGraph
2. 实现 Supervisor 模式，动态决定下一步

**长期（2月）**:
1. 支持循环与迭代
2. Human-in-the-Loop 中断点

- 上下文通过 `context` 字典在 Agent 间传递（`orchestrator.py` 89行）

❌ **缺失能力对比**:

| 能力 | 行业标杆 | 本系统 |
|------|----------|--------|
| Agent 间 Handoff（任务移交） | OpenAI Agents SDK | ✗ |
| 结构化消息协议 | AutoGen (ChatMessage) | ✗ 仅字符串拼接 |
| 共享状态存储 | LangGraph (State) | ✗ 仅传 dict |
| 对话历史管理 | 所有主流框架 | ✗ 每次只传单条消息 |
| 广播/订阅模式 | AutoGen (GroupChat) | ✗ |
| 消息过滤/路由 | CrewAI | ✗ |

#### 问题所在：字符串拼接式通信

**代码证据** (`orchestrator.py` 125-136行):
```python
content=(
    f"任务：{task}\n\n规划：\n{plan_resp.content}\n\n"
    f"Worker 输出：\n{worker_outputs}"  # ← 这里 worker_outputs 是 list 直接 str()
)
```

**核心问题**:
1. **非结构化**: Reviewer 看到的是 Python list 的 `repr` 输出，如 `['结果1', '结果2']`，而不是结构化数据
2. **信息丢失**: Worker 的元数据（执行时间、使用的工具、中间步骤）全部丢失
3. **无法追溯**: 无法知道某个结论来自哪个 Worker
4. **无历史**: Agent 之间没有共享的对话历史，每次都是全新对话

**实际案例**:

当前系统输出给 Reviewer 的内容：
```
任务：分析市场趋势

规划：
1. 搜索最新资料
2. 分析数据
3. 撰写报告

Worker 输出：
['Worker1找到了5篇文章', 'Worker2发现增长率20%', 'Worker3完成报告草稿']
```

行业标杆（LangGraph）的做法：
```python
# 共享状态对象
class State(TypedDict):
    task: str
    plan: Plan
    research_results: List[ResearchItem]  # 结构化数据
    analysis: Analysis  # 结构化数据
    report_draft: str
    review_feedback: Optional[str]

# 每个 Agent 读写这个状态
# Reviewer 可以访问完整的结构化信息
```

#### 改进建议

**短期（1周）**:
1. 将 `worker_outputs` 从 list 改为结构化对象列表
2. 传递完整的 WorkerResult（含 worker_name, content, metadata）

**中期（3周）**:
1. 实现共享状态对象（TypedDict），所有 Agent 读写
2. 建立消息总线，支持类型化消息

**长期（2月）**:
1. 实现对话历史管理
2. Handoff 机制（参考 OpenAI Agents SDK）

#### 问题所在：Agent 完全无记忆

**代码证据** (`agent.py` 114-119行):
```python
def _build_messages(self, message: AgentMessage) -> List[Dict[str, str]]:
    user_content = self._build_user_content(message)
    return [
        {"role": "system", "content": self.system_prompt},
        {"role": "user", "content": user_content}
    ]
    # ← 只有两条消息！没有历史对话
```

**核心问题**:
1. **无短期记忆**: 每次调用 LLM 只传 `[system, user]` 两条消息
2. **无法多轮对话**: Agent 看不到之前的对话历史
3. **MEMORY/ 目录虚设**:
   - `MEMORY/tasks/` 仅存储任务执行记录（状态追踪），无检索能力
   - `MEMORY/logs/` 只有手写开发日志
   - `raw_data/`、`structured/`、`reports/` 目录在 README 中提到但未实现
4. **无向量存储**: 没有集成 ChromaDB/FAISS/Pinecone

**实际问题场景**:
```
用户第1轮：请帮我分析一下 Python 的市场趋势
Agent：[分析完成，给出结果]

用户第2轮：刚才你提到的增长率数据来源是什么？
Agent：[无法回答，因为 Agent 已经忘记第1轮的对话内容]
```

**行业标杆做法**:
```python
# CrewAI 的四层记忆
class Agent:
    short_term_memory: List[Message]  # 当前会话
    long_term_memory: SQLite          # 跨会话持久化
    entity_memory: ChromaDB           # 向量化的实体记忆
    context_memory: Dict              # 当前上下文
```

#### 改进建议

**短期（2周）**:
1. Agent 维护 message history，多轮对话时传递完整历史
2. 实现 token 计数，超限时做滑动窗口

**中期（1月）**:
1. 集成向量存储（ChromaDB/FAISS）
2. 将 `MEMORY/` 目录升级为可查询的知识库

**长期（2月）**:
1. 实现摘要压缩策略

#### 问题所在：原始的关键词匹配

**代码证据** (`agent.py` 73-78行):
```python
content = message.content.lower()
for skill_name in self.skills:
    keywords = router.get_skill_keywords(skill_name)
```

当前系统：因为包含"搜索"二字，触发 web_search 技能 ❌

行业标杆（OpenAI Agents SDK）:
```python

tools = [
    {
        "type": "function",
        "function": {
            "name": "web_search",
            "description": "在网络上搜索信息",
            "parameters": {
                "type": "object",
                "properties": {
                    "query": {"type": "string"}
                },
                "required": ["query"]
            }
        }
    }
]
# LLM 看到用户说"不要使用搜索"，会选择不调用这个工具 ✓
```

## 四、核心风险提示

### 🚨 严重风险

1. **零测试覆盖是最大的系统性风险**
   任何重构都可能引入无法察觉的回归 Bug。建议在任何功能开发之前，先建立测试基线。

2. **关键词路由的误触发风险**
   用户输入中偶然包含"搜索"、"数据"等关键词就会触发错误的技能调用。必须切换到 LLM Function Calling。

3. **无并发保护**
   `TaskTracker` 使用文件系统持久化，多进程/多线程并发写入同一个 JSON 文件会导致数据损坏。需要引入文件锁或换用数据库。

4. **同步 HTTP 调用的性能瓶颈**
   `ai_client.py` 的 HTTP SDK 使用 `requests.post()`（同步阻塞），在多 Worker 并行场景下，`asyncio.to_thread` 只是将阻塞移到线程池，无法真正利用异步 I/O 优势。
