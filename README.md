  一个面向个人开发者的、可持续进化的多 Agent 架构系统。用户通过不断积累和调整 Skills，使系统逐渐成为符合个人使用习惯的专属 AI 工作站。更换 LLM API Key 后，积累的 Skills、偏好、经验不丢失。

4.Skills加载逻辑非常”智能“，实践中反而过度工程化

核心错误：在单Agent抵达能力边界前，试图通过多Agent来拓宽能力范围


核心经验总结：
1.从第一性原理思考问题，从具体的场景/任务出发。
2.在出现具体瓶颈时思考”该加什么东西“，而不是在问题出现前就过度工程化。
3.让现有框架处理基础设施，不是把研究如何“造轮子”。
4.架构需要服务于问题，架构不是目的本身
5.让 Agent 的复杂性来自于 LLM 本身的能力，而不是围绕它搭的脚手架。


# AgentSystem (archived)

> 状态：已存档。  
> 本仓库保留了我在 Agent 编排方面的一次失败尝试，以纪念我第一次以“项目落地”为目标的 Vibe Coding 经历，记录和反思经验。

---

## 项目初衷

基于 Anthropic 的 Skills 渐进式披露机制，加上一套可以持续演化的架构，尝试搭建适合个人长期使用的多 Agent 系统。

- 使用 CLI Agent（如 Claude Code、Codex 等）作为系统调用工具  
- 可以按需调度不同 Agent，完成其各自擅长领域的任务  
- 随着 LLM 能力进步，系统整体能力也随之提升

---

## 项目愿景

面向个人开发者的、可持续进化的多 Agent 架构系统。

- 通过不断积累和调整 Skills，让系统逐渐变成符合个人习惯的专属 AI 工作站  
- 更换 LLM API Key 后，既有的 Skills、偏好、经验不丢失  
- 架构与具体模型解耦，把“经验”和“知识”留在系统里

---

## 设计时具备的特性

设计方案中具有完整的“操作系统式”组件：

- 路由与编排器（Router & Orchestrator）  
- 规划者 / 执行者 / 评审者（Planner / Worker / Reviewer）角色模型  
- 用于统一输入/输出协议的 5 个 dataclass（ExecutionInput / SkillInput / ExecutionOutput / SkillOutput / AgentMessage 等）  
- 4 层技能加载回退机制（注册表 → entrypoint → `SKILL.md` → `skill.py`）  
- 自定义的 TaskTracker 持久化与错误分类体系


---

## 问题总结

1. **协议设计超前**  
   在跑通 10 个真实工作流之前，就设计可以支撑 200 个工作流的统一协议和抽象层。

2. **过早引入多角色模型**  
   在单 Agent 还没有真正发挥出价值时，就引入 多角色，试图通过结构复杂度来提升系统能力。

3. **可观测性优先于可用性**  
   在日志、追踪器和错误分类上投入大量精力，但端到端任务本身并不稳定，真实“完成一个任务”的过程依然是黑盒。

4. **Skills 加载逻辑过于“智能”**  
   设计了 4 层 fallback 的技能加载机制，实践中只用到 1–2 层，复杂度变成了维护负担。

**核心错误：**

> 在单 Agent 还未抵达能力边界前，就试图通过多 Agent 和复杂编排来“补齐”能力。

---

## 核心经验总结

这次尝试改变了我对 Agent 系统理解：

1. **从具体场景和任务出发，用第一性原理思考问题**  
   先问清楚：要解决的具体任务是什么？完成它最少需要哪些能力和组件？

2. **在真实瓶颈出现时再“加东西”**  
   而不是在问题出现之前，预先设计大量抽象和扩展点，避免过度工程化。

3. **优先使用成熟框架处理基础设施**  
   状态管理、持久化、编排等基础设施尽量交给现有框架（如 LangGraph 等），把精力放在行为逻辑和 Skills ，而不是一开始就研究“造轮子”。

4. **架构服务于问题，架构不是目的本身**  
   先确保有 Agent 能稳定解决一个真实问题，再考虑如何抽象和扩展；而不是先搭架构，再去找问题来填。

5. **让复杂性停留在 LLM，而不是脚手架**  
   让 Agent 变强不是围绕它堆砌的大量中间层和控制结构。
